# Use Cases – EventForge (Платформа для организации мероприятий)

## 1. Регистрация нового пользователя (Организатор / Участник)

**Предусловия:**
Пользователь впервые открывает приложение.

**Триггер:**
Пользователь нажимает кнопку «Зарегистрироваться».

**Основной сценарий:**
1. Система предлагает выбрать тип аккаунта: «Организатор» или «Участник».
2. Система предлагает ввести email (или номер телефона) и пароль.
3. Пользователь вводит email (или номер телефона) и пароль, соответствующий требованиям безопасности.
4. Система проверяет, что email (или номер телефона) еще не зарегистрирован.
5. Система отправляет шестизначный код подтверждения на указанный email (или номер телефона).
6. Пользователь вводит полученный код в соответствующее поле.
7. Система подтверждает код и предлагает заполнить профиль (имя, город, интересы).
8. Пользователь вводит имя и другие данные.
9. Система перенаправляет пользователя на главный экран.

**Результат:**
Создан персональный аккаунт пользователя.

**Альтернативный сценарий:**
* **4a. Email занят:** Если email уже занят, система сообщает: «Аккаунт с таким email уже существует. Попробуйте войти или восстановить пароль».
* **6a. Неверный код:** Если код введен неверно, система отображает ошибку: «Неверный код подтверждения».

## 2. Создание мероприятия

**Предусловия:**
Пользователь авторизован как организатор.

**Триггер:**
Пользователь нажимает кнопку «Создать мероприятие» на главном экране.

**Основной сценарий:**
1. Система открывает форму создания мероприятия с полями: название, формат (онлайн / оффлайн / гибрид), город, дата, время, количество гостей, бюджет.
2. Пользователь начинает заполнять основные поля.
3. Интеллектуальный помощник анализирует введённые данные и в реальном времени:
    * подсказывает оптимальную длительность мероприятия,
    * рекомендует подходящий формат,
    * предлагает примерную структуру (регистрация, основная программа, перерывы).
4. Пользователь принимает или отклоняет подсказки.
5. Пользователь заполняет остальные поля.
6. Пользователь нажимает кнопку «Опубликовать».
7. Система сохраняет мероприятие и публикует его в системе.

**Результат:**
В системе создано новое мероприятие с корректно заполненными параметрами.

**Альтернативный сценарий:**
* **3a. Игнорирование ИИ:** Пользователь игнорирует подсказки ИИ и заполняет форму полностью вручную.
* **6a. Не заполнены поля:** Если обязательные поля не заполнены, система выводит сообщение: «Заполните все обязательные поля».

## 3. Гибкий конструктор программы мероприятия

**Предусловия:**
* Пользователь авторизован.
* В системе существует созданное мероприятие.

**Триггер:**
Пользователь заходит в карточку мероприятия и нажимает «Программа».

**Основной сценарий:**
1. Система открывает редактор программы мероприятия.
2. Пользователь добавляет временные блоки (доклады, шоу, перерывы).
3. Для каждого блока указывает название, описание, ведущего, время начала и длительность.
4. Пользователь изменяет порядок блоков с помощью Drag & Drop.
5. Пользователь нажимает «Сохранить».
6. Система валидирует временные интервалы и сохраняет программу.

**Результат:**
Создана структурированная программа мероприятия.

**Альтернативный сценарий:**
* **6a. Пересечение времени:** Если блоки времени пересекаются, система показывает ошибку: «Обнаружено пересечение временных интервалов».

## 4. Создание и продажа билетов с уровнями доступа

**Предусловия:**
Мероприятие опубликовано.

**Триггер:**
Организатор нажимает кнопку «Настроить билеты».

**Основной сценарий:**
1. Система открывает форму создания билетов.
2. Пользователь создаёт типы билетов (Стандарт, VIP, Онлайн).
3. Указывает стоимость, количество, период продаж.
4. Пользователь нажимает «Сохранить».
5. Участники покупают билеты через приложение.
6. После покупки билет сохраняется в профиле участника.

**Результат:**
Настроена и запущена продажа билетов.

**Альтернативный сценарий:**
* **5a. Билеты закончились:** Если количество билетов исчерпано, система автоматически завершает продажи.

---

## 5. Регистрация участников на входе по QR-коду

**Предусловия:**
Пользователь приобрёл билет.

**Триггер:**
Пользователь прибывает на мероприятие.

**Основной сценарий:**
1. Пользователь открывает билет в приложении.
2. Показывает QR-код сотруднику мероприятия.
3. Система сканирует код.
4. Система проверяет валидность билета.
5. Пользователь отмечается как присутствующий.

**Результат:**
Пользователь успешно зарегистрирован на входе.

**Альтернативный сценарий:**
* **4a. Недействительный код:** Если QR-код недействителен, система блокирует вход и уведомляет администратора.

---

## 6. Управление бюджетом мероприятия

**Предусловия:**
* Пользователь авторизован.
* Создано мероприятие.

**Триггер:**
Пользователь открывает раздел «Финансы».

**Основной сценарий:**
1. Пользователь указывает общий бюджет мероприятия.
2. Добавляет статьи расходов (площадка, реклама, техника).
3. Вносит фактические траты.
4. Система автоматически рассчитывает остаток бюджета.
5. При превышении лимита система отображает предупреждение.

**Результат:**
Пользователь видит актуальное финансовое состояние мероприятия.

**Альтернативный сценарий:**
* **1a. Бюджет не задан:** Если бюджет не задан, система предлагает создать шаблон бюджета.

## 7. Командная работа организаторов

**Предусловия:**
* Пользователь авторизован.
* Создано мероприятие.

**Триггер:**
Пользователь нажимает «Добавить участника команды».

**Основной сценарий:**
1. Система предлагает ввести email пользователя.
2. Пользователь назначает роль (администратор, координатор, модератор).
3. Пользователь нажимает «Отправить приглашение».
4. Приглашённый пользователь получает уведомление.
5. После подтверждения получает доступ к мероприятию.

**Результат:**
К мероприятию подключён новый участник команды.

**Альтернативный сценарий:**
* **5a. Отклонение приглашения:** Если приглашение отклонено, система не предоставляет доступ.

---

## 8. Умный нетворкинг участников (AI-подбор контактов) 

**Предусловия:**
Пользователь зарегистрирован на мероприятие.

**Триггер:**
Пользователь заходит в раздел «Нетворкинг».

**Основной сценарий:**
1. Пользователь указывает цель участия (бизнес, партнёрство, знакомства).
2. Система анализирует профили участников (сфера деятельности, интересы).
3. ИИ формирует список релевантных контактов.
4. Пользователь выбирает участника и отправляет запрос.
5. После согласия открывается чат.

**Результат:**
Пользователь получает доступ к релевантным контактам.

**Альтернативный сценарий:**
* **3a. Совпадений нет:** Если совпадения не найдены, система отображает общий список участников.

---

## 9. Массовые уведомления участникам

**Предусловия:**
Существуют зарегистрированные участники.

**Триггер:**
Организатор нажимает кнопку «Отправить уведомление».

**Основной сценарий:**
1. Организатор вводит текст сообщения.
2. Выбирает получателей (все / VIP / спикеры).
3. Нажимает «Отправить».
4. Система отправляет push-уведомления.

**Результат:**
Участники оперативно получают сообщение.

**Альтернативный сценарий:**
* **4a. Нет интернета:** При отсутствии интернета сообщение доставляется при следующем входе.

---

## 10. Формирование итогового отчёта по мероприятию

**Предусловия:**
Мероприятие завершено.

**Триггер:**
Организатор нажимает кнопку «Сформировать отчёт».

**Основной сценарий:**
1. Система собирает статистику (посещаемость, билеты, финансы).
2. Система формирует итоговый отчёт в формате PDF.
3. Пользователь нажимает «Скачать».

**Результат:**
Сформирован официальный аналитический отчёт мероприятия.

**Альтернативный сценарий:**
* **1a. Нехватка данных:** Если не хватает данных, система сообщает, какие параметры не заполнены.
